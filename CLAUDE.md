## Commands

- `npm run dev` — Next dev server (Turbopack)
- `npm run build` / `npm run start` — production build / serve
- `npm run lint` — ESLint (flat config, `eslint-config-next/core-web-vitals` + `typescript`)
- `npm run db:push` — apply `src/lib/schema.ts` directly to Neon via `drizzle-kit push`. There are no migration files — schema changes are pushed live. `.env.local` is auto-loaded by the script.

There is no test suite.

## Next.js 16 specifics that bite

- **`src/proxy.ts`** is the Next.js 16 replacement for `middleware.ts` — same `NextRequest`/`NextResponse` API, same `config.matcher`, but the file name and exported function name (`proxy`) are different. Don't rename it to `middleware`. See `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/proxy.md`.
- `cookies()` and similar request APIs are async — `await cookies()` (see `src/lib/auth.ts`).
- Drizzle's `onConflictDoUpdate` here uses `setWhere` to scope the update to the owning user (`src/app/api/todos/route.ts`) — preserve that when editing upsert handlers.

## Architecture

### Two databases, one source of truth

The app is offline-first. Data lives in **two stores** and they are reconciled, not mirrored:

- **Local (`src/lib/db.ts`)** — Dexie/IndexedDB. Each row carries a `sync_status` of `synced | pending | deleting`. Schema is versioned with explicit `db.version(N).stores({...}).upgrade(...)` blocks; bump the version and add an upgrade when changing local shape.
- **Server (`src/lib/db-server.ts` + `src/lib/schema.ts`)** — Neon Postgres via Drizzle. The server is the canonical copy.

Reconciliation lives in **`src/lib/sync.ts`** and runs from `SyncRunner` on mount, every 30s, and on `visibilitychange`. The order is: verify session → reset local store if `user_id` changed → `pushDirty` (pending/deleting todos) → `pullAll` (last-write-wins on `updated_at`, deletes any local `synced` row missing from the server) → tags + todo_tags → prefs. When you add a new synced entity, follow this same push-pending → pull-and-reconcile pattern and clear it in the user-switch reset block.

Mutation helpers in `src/lib/todos.ts` / `src/lib/tags.ts` / `src/lib/prefs.ts` write locally with `sync_status: "pending"` (or `"deleting"`) and let the next sync flush. UI components should call those helpers, not `fetch` directly.

### Auth

- `src/proxy.ts` redirects unauthenticated browser requests to `auth.andrewcromar.org/pages/login.html?redirect=...`. The matcher excludes `/api/`, `_next/`, the service worker, and static assets — API routes do their own auth.
- `src/lib/auth.ts` `getUser()` validates `session_token` against `${NEXT_PUBLIC_AUTH_BASE_URL}/api/validate_token.php` server-side. Every API route that touches user data must call `getUser()` and 401 on null; mutating routes additionally check `existing.user_id === user.id` and 403 on mismatch.
- The session cookie is set by web_auth on `.andrewcromar.org` so it's shared across subdomains. Local dev against XAMPP needs `NEXT_PUBLIC_AUTH_BASE_URL=http://localhost/web_auth/public`.

### Push notifications

- Subscriptions are stored in `push_subscriptions` (`src/lib/schema.ts`).
- Two cron endpoints under `src/app/api/cron/` (`reminders`, `daily-digest`) are triggered externally (cron-job.org). They auth via `CRON_SECRET` accepted in three forms: `Authorization: Bearer <secret>`, `x-cron-secret` header, or `?secret=` query param. Keep all three when adding new cron routes.
- `reminders` uses `notified_at` as the dedupe flag and clears it server-side on `due_at` change (see the `dueChanged` branch in `src/app/api/todos/route.ts`).
- The service worker is **hand-written at `public/sw.js`**, not generated by `@serwist/next` despite serwist being a dep. Edit it directly; `SyncRunner` calls `registration.update()` on every load to force refresh.

### Environment

Required in `.env.local` (and Vercel project settings):

```
DATABASE_URL                    # Neon Postgres
NEXT_PUBLIC_AUTH_BASE_URL       # web_auth origin
NEXT_PUBLIC_VAPID_PUBLIC_KEY    # web-push (also exposed to client)
VAPID_PRIVATE_KEY               # web-push
VAPID_SUBJECT                   # mailto: or https: identifying owner
CRON_SECRET                     # shared with cron-job.org
```
